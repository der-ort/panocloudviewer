# Data Format

PanoCloudViewer expects data in the **Potree 2.0** format. This is the standard output of [PanoCloudConverter](/guide/point-cloud-preparation) and compatible with any Potree 2.0 pipeline.

---

## Folder Structure

```
project/
├── metadata.json       # Potree 2.0 octree metadata (required)
├── hierarchy.bin       # Octree hierarchy (required)
├── octree.bin          # Point cloud data (required)
└── cameras.json        # Panorama camera positions (optional)
```

All files must be accessible via the configured [source adapter](/guide/getting-started#source-types). For S3, the entire folder lives under a common base URL.

---

## metadata.json

Standard Potree 2.0 metadata file generated by the converter. It describes the octree structure, bounding box, point attributes (XYZ, RGB, intensity, classification, etc.), and scale/offset values.

No manual editing is required — this file is produced automatically.

---

## cameras.json

An optional JSON array describing panoramic camera positions. When present, the viewer renders interactive camera markers in the 3D scene and enables the [panorama viewer](/features/panoramas).

### Schema

```typescript
interface CameraData {
  /** Display name / filename of the image */
  name: string;

  /** Zero-based ordering index */
  index: number;

  /** URL of the panorama image (JPEG, PNG) */
  image: string;

  /** 3D position in the point cloud coordinate system */
  position: { x: number; y: number; z: number };

  /** Horizontal heading in degrees (0 = North / +Y axis) */
  yaw_deg: number;

  /** Panorama type */
  representation:
    | 'sphericalRepresentation'
    | 'pinholeRepresentation'
    | 'cylindricalRepresentation';

  /** Optional quaternion rotation */
  rotation?: { x: number; y: number; z: number; w: number };

  /** Optional human-readable description */
  description?: string;

  /** Source of the position ('scan' | 'gps' | 'manual') */
  position_source?: string;
}
```

### Example

```json
[
  {
    "name": "IMG_0001",
    "index": 0,
    "image": "https://cdn.example.com/panoramas/IMG_0001.jpg",
    "position": { "x": 14.03, "y": -5.85, "z": 528.18 },
    "yaw_deg": 45.0,
    "representation": "sphericalRepresentation",
    "rotation": { "x": 0, "y": 0, "z": 0, "w": 1 },
    "description": "Entrance lobby",
    "position_source": "scan"
  },
  {
    "name": "IMG_0002",
    "index": 1,
    "image": "https://cdn.example.com/panoramas/IMG_0002.jpg",
    "position": { "x": 20.11, "y": -3.42, "z": 528.22 },
    "yaw_deg": 90.0,
    "representation": "sphericalRepresentation"
  }
]
```

## S3 CORS Configuration

When serving data from S3, configure the bucket CORS policy to allow range requests:

```json
[
  {
    "AllowedOrigins": ["*"],
    "AllowedMethods": ["GET", "HEAD"],
    "AllowedHeaders": ["*"],
    "ExposeHeaders": ["Content-Range", "Accept-Ranges"],
    "MaxAgeSeconds": 3600
  }
]
```

The `Content-Range` and `Accept-Ranges` headers are required for partial loading of large octree files.

---

## Coordinate System

The viewer uses the coordinate system embedded in `metadata.json`. The `position` values in `cameras.json` must be in the **same coordinate system** as the point cloud.

If your scanner exports in a geographic coordinate system (e.g. EPSG:25832), the converter handles the offset and scale automatically — no manual reprojection needed.
